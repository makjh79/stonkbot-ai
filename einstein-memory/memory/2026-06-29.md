# Alpaca 403 Sell Bug — Full Investigation (2026-06-29 ~17:15 UTC)

## Initial symptoms
- System logs showed repeated 403 errors on sell order submissions: `Failed to submit sell order for {symbol}: 403 Client Error: Forbidden`.
- Occurred every ~5 minutes during market hours (bot cycle interval).
- Multiple symbols affected: AAPL, AMZN, NVDA, COST, ABNB, AMD, SHOP, MU, UPST, SOFI, etc.
- Monitor flagged `signals.json` stale (~15 min) at one point, but later recovered.

## Diagnostic steps taken
1. Verified Alpaca API credentials in `/opt/stonk-ai/alpaca_config.json`:
   - `GET /v2/account` → 200 (ACTIVE, paper account)
   - `GET /v2/positions` → 200
   - Direct `POST /v2/orders` market BUY of 1 AAPL → 200 (filled)
   - Direct `POST /v2/orders` market SELL of 1 AAPL → 200 (filled)
   - **Conclusion: API keys are valid and have trading scope.**
2. Direct `POST /v2/orders` LIMIT SELL of 1 AAPL:
   - Returned **403**: `{"code":40310000,"message":"insufficient qty available for order (requested: 1, available: 0)"}`
   - Response also included: `"existing_qty":"10","held_for_orders":"10","related_orders":["aabe07b0-486f-4c85-86e5-f58d9d46ab1d"]`
3. Queried open orders on paper account:
   - Exactly **1 open order**: `AAPL sell limit qty=10 price=285.92` submitted 13:39 UTC, status `new`, filled_qty=0.
   - This order had been sitting unfilled for **~4 hours**.
4. Queried all-order history for the day:
   - 122 sell orders placed in total.
   - Most filled successfully.
   - Only AAPL remained stuck.

## Root cause(s)
### 1. Stuck limit sell order
- `trading_bot.py` computes exit price as `limit_price = round(midpoint, 2)` via `get_latest_quote()` at the moment the trailing/hard stop triggers.
- AAPL breached its trailing stop at ~$286 midpoint, generating a limit sell @ $285.92.
- The stock continued dropping to ~$281. A sell limit **above** the market price will never fill in a declining market.

### 2. Missing open-order deduplication guard
- `trading_bot.py` (line 1361 area) calls `self.alpaca.submit_order(...)` inside `_execute_sell` **without first checking** whether an open sell order already exists for that symbol.
- Every 5-minute cycle, the bot re-evaluates the trailing stop, sees the AAPL position is still held, and tries to sell again.
- Alpaca rejects the duplicate with 403 because all 10 AAPL shares are already locked/committed to the existing open order.
- This produced the repeating error log flood for AAPL (and for other symbols earlier in the day that had similar transient unfilled orders before they eventually cleared or filled).

### 3. Limit orders for stop exits are risky in fast markets
- Using limit orders for stop-loss/trailing-stop exits can leave the bot with unfilled orders while the price continues to fall.
- Market orders would guarantee execution but at the cost of slippage. A hybrid approach (e.g., limit → market after a timeout, or bracket orders) might be safer.

- **BOT_DIR missing:** `TradingConfig` class in `trading_bot.py` lacked `BOT_DIR` attribute, causing `Could not log trade: type object 'TradingConfig' has no attribute 'BOT_DIR'`. Added `BOT_DIR: Path = Path(__file__).parent`.
- **regime_status.json permissions:** file was owned by `root`, causing permission denied writes from the `stonkai` user service. Changed ownership to `stonkai:stonkai`, mode `664`.
- **signals.json freshness:** confirmed file is actually current (updated 17:19 UTC). Monitor stale flag was transient.

2. **Patch order deduplication guard** — before `submit_order('sell')`, check `list_orders(status='open', symbols=[symbol])` for an existing sell order and skip/return if found.
3. **Consider market-order exits for stops** — to guarantee fills on breached stops rather than relying on stale midpoints.
